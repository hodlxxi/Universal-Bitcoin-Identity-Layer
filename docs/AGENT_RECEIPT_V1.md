# HODLXXI Agent Receipt v1

HODLXXI agent receipt v1 is the public contract for proving that a Lightning-paid agent job completed under the HODLXXI paid agent commerce runtime. It documents the existing runtime only; it does not introduce new payment, job, receipt, or verification behavior.

## Purpose

A receipt binds one completed agent job to:

- the original request hash,
- the Lightning invoice/payment hash recorded for that job,
- the produced result hash,
- the HODLXXI agent public key, and
- the agent signature over the receipt payload.

External clients should treat `/agent/jobs/<job_id>` as the lifecycle/status endpoint and `/agent/verify/<job_id>` as the receipt verifier after a receipt exists.

## Lifecycle

1. Discover public surfaces with `GET /.well-known/agent.json`, `GET /agent/capabilities`, or `GET /agent/marketplace/listing`.
2. Create a paid job with `POST /agent/request`.
3. The runtime creates a Lightning invoice through the configured `LN_BACKEND` such as `lnd_cli` in production, stores an `AgentJob`, and returns `status=invoice_pending` with an invoice and `payment_hash`.
4. The operator or external tester performs manual payment. HODLXXI smoke/runbook flows must never auto-pay invoices.
5. The client polls `GET /agent/jobs/<job_id>`, the lifecycle/status endpoint.
6. After payment is observed by Lightning invoice checking, the runtime executes the job, stores the result, signs a `job_receipt`, and appends an `AgentEvent` attestation.
7. `GET /agent/verify/<job_id>` verifies the issued receipt and returns `status=verified` and `valid=true` when the signature verifies.
8. `GET /agent/attestations` exposes signed receipt events for completed jobs, and `GET /agent/reputation` aggregates operating history from jobs and attestations.

## Runtime relationships

- `AgentJob` stores job identity, `job_type`, request JSON, `request_hash`, sat price, Lightning invoice, payment lookup id, `payment_hash`, lifecycle status, result JSON, and `result_hash`.
- `AgentEvent` stores signed attestation log entries, including the `event_hash`, `prev_event_hash`, receipt JSON, and signature.
- `payment_hash` is derived from the Lightning invoice lookup id used by the runtime. With `LN_BACKEND=lnd_cli`, the lookup id is the LND invoice `r_hash` returned by `lncli addinvoice`; the stored public `payment_hash` is the SHA-256 hex hash of that lookup id in the current code path.
- `request_hash` is SHA-256 over canonical JSON for `{"job_type": ..., "payload": ...}`.
- `result_hash` is SHA-256 over canonical JSON for the result payload produced after payment.
- The receipt is also published as a `job_receipt` attestation and contributes to reputation/marketplace trust surfaces.

## Receipt fields

The current v1 receipt is a JSON object with these required/expected fields:

| Field | Meaning |
| --- | --- |
| `version` | Receipt contract version. Current runtime uses `1.0`. |
| `event_type` | Receipt event type. Current runtime uses `job_receipt`. |
| `job_id` | UUID of the `AgentJob`. |
| `job_type` | Job type requested through `/agent/request`, such as `ping`. |
| `payment_hash` | Public payment linkage for the invoice-backed job. |
| `request_hash` | Hash of the canonical request payload. |
| `result_hash` | Hash of the canonical result payload. |
| `prev_event_hash` | Previous attestation event hash, or `null` for the first event. |
| `timestamp` | Receipt issuance timestamp. |
| `agent_pubkey` | Compressed secp256k1 public key for the HODLXXI agent signer. |
| `signature` | ECDSA/SHA-256 signature generated by the agent signer. |

## Canonical JSON and signature coverage

The existing signer serializes JSON with sorted keys and compact separators before hashing/signing. Receipt verification removes the `signature` field and verifies the signature over the remaining canonical JSON payload. The receipt `event_hash` is computed as SHA-256 over the canonical JSON receipt including the `signature` field.

Do not assume additional cryptographic claims beyond what the current code enforces: the signature covers the receipt payload fields, and the runtime verifier checks that signature against `agent_pubkey`.

## Verification response fields

`GET /agent/verify/<job_id>` is the receipt verifier. After a receipt exists, the response includes:

| Field | Meaning |
| --- | --- |
| `job_id` | Job id being verified. |
| `status` | `verified` when the receipt signature is valid; otherwise an error/invalid status. |
| `valid` | Boolean signature verification result. |
| `event_hash` | SHA-256 hash of the canonical receipt event. |
| `receipt` | Stored receipt JSON. |
| `attestation` | Normalized attestation view of the receipt event. |
| `agent_pubkey` | Public key used for verification. |

Expected paid state is `status=done` on `/agent/jobs/<job_id>`, with `result` present, `receipt` present, and `/agent/verify/<job_id>` returning `status=verified` and `valid=true`.

## Unpaid/no-receipt semantics

An unpaid job may exist while no receipt exists yet:

- `/agent/jobs/<job_id>` remains the lifecycle/status endpoint.
- The job can return `status=invoice_pending`, `result=null`, and `receipt=null`.
- `/agent/verify/<job_id>` verifies receipts only after issuance.
- Before receipt issuance, `/agent/verify/<job_id>` may return `404` with `verification=unavailable`, even when the job exists and is `invoice_pending`.
- `/agent/attestations` should not contain a `job_receipt` event for an unpaid job.

## Attestation chain behavior

When the runtime issues a receipt, it appends an `AgentEvent`. Each new event records `prev_event_hash` as the previous chain head, and `/agent/chain/health` can check the observed chain order. This is an append-only attestation log behavior in the current application; clients should still verify each receipt signature and should not overclaim global consensus or external anchoring from the local event chain alone.
